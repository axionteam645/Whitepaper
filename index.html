Executive Summary

Code-verified overview of AXION as a minimal, fixed-supply, control-free ERC-20 token deployed on BNB Smart Chain.

AXION is a minimal ERC-20 token implementation deployed on BNB Smart Chain, designed with a fixed supply and a simplified, deterministic execution model.

The contract intentionally limits its functionality to core ERC-20 operations, avoiding additional modules, dynamic parameters, or administrative control mechanisms.

1.1 Fixed Supply Model

AXION operates with a fixed total supply, defined at deployment and not subject to change.

Code Proof
uint256 public constant MAX_SUPPLY = 1_000_000_000 * 10**18;
uint256 public totalSupply = MAX_SUPPLY;

Technical Interpretation

  • MAX_SUPPLY is defined as a constant value
  • totalSupply is initialized once
  • No function exists that modifies totalSupply

Summary

The supply model is static and does not expose inflation, emission, or post-deployment supply expansion paths.

1.2 Single Genesis Distribution

All tokens are assigned at deployment through a single initialization event.

Code Proof

  • Constructor executes only once
  • Entire supply is transferred from zero address

1.3 Standard ERC-20 Functionality

The contract supports only standard ERC-20 operations.

Code Proof

function transferFrom(address from, address to, uint256 amount) external returns (bool)

1.4 Deterministic Transfer Logic

All token transfers are executed through direct balance updates without additional logic layers.

Code Proof
function _transfer(address from, address to, uint256 amount) internal {
    require(from != address(0), "Invalid sender");
    require(to != address(0), "Invalid recipient");
    require(balanceOf[from] >= amount, "Insufficient balance");

    balanceOf[from] -= amount;
    balanceOf[to] += amount;

    emit Transfer(from, to, amount);
}
  • Transfers update balances directly
  • No fees or deductions are applied
  • Execution is linear and predictable

System Architecture

Direct on-chain state architecture built around transparent mappings, deterministic execution, and event-based traceability.

AXION utilizes a direct on-chain state architecture, where all token data is stored and managed through deterministic storage mappings defined within the smart contract.

The system avoids complex data structures, external dependencies, and modular extensions, resulting in a simple, transparent, and fully auditable state model.

2.1 On-Chain State Model

The AXION contract maintains its entire state using two primary mappings: balanceOf for token ownership and allowance for delegated spending permissions.

Code Proof
mapping(address => uint256) public balanceOf;
mapping(address => mapping(address => uint256)) public allowance;
  • balanceOf stores the token balance of each address
  • allowance defines how much a third party can spend on behalf of a token holder
  • Both mappings are stored directly on-chain
  • No external storage or off-chain dependency exists

2.2 Execution Model

All state transitions in AXION are executed synchronously within the EVM, without any asynchronous operations or external contract interactions.

  • All functions execute atomically
  • State changes are applied instantly within a single transaction
  • If any condition fails, the entire transaction reverts

This guarantees that no partial state changes occur, no race conditions exist, and execution remains predictable under all conditions.

2.3 Event Layer (State Transparency)

The contract emits standard ERC-20 events to ensure full traceability of all token interactions.

Code Proof
event Transfer(address indexed from, address indexed to, uint256 value);
event Approval(address indexed owner, address indexed spender, uint256 value);
  • Transfer event records all token movements
  • Approval event records allowance updates
  • Events are indexed for efficient tracking

2.4 Internal State Isolation

AXION does not interact with external contracts, oracles, or off-chain systems.

Verified Behavior

  • No external contract calls (call, delegatecall, etc.)
  • No oracle integrations
  • No dependency injection

2.5 Deterministic Data Flow

All data flows within AXION follow a strictly defined path: input → validation → state update → event emission.

Example Flow

2.6 Absence of Dynamic State Logic

Allowance & Delegation

Standard ERC-20 delegation logic enabling controlled third-party token spending with explicit approval and deterministic allowance reduction.

AXION implements a standard ERC-20 allowance model, enabling token holders to authorize third-party addresses to spend tokens on their behalf through a controlled and deterministic delegation mechanism.

This system allows integration with decentralized exchanges, liquidity pools, and automated protocols while maintaining strict on-chain validation.

5.1 Approval Mechanism

Token holders can assign spending permissions using the approve function.

Code Proof
function approve(address spender, uint256 amount) external returns (bool) {
    require(spender != address(0), "Invalid spender");

    allowance[msg.sender][spender] = amount;
    emit Approval(msg.sender, spender, amount);

    return true;
}
  • The caller (msg.sender) defines the allowance
  • The spender is authorized to use a specified amount
  • The allowance value is directly set and overwrites the previous value
  • An Approval event is emitted for transparency

5.2 Allowance Storage Model

Code Proof
mapping(address => mapping(address => uint256)) public allowance;
  • First key: token owner
  • Second key: authorized spender
  • Value: maximum spendable amount

5.3 Delegated Transfer Execution

Code Proof
function transferFrom(address from, address to, uint256 amount) external returns (bool) {
    uint256 allowed = allowance[from][msg.sender];
    require(allowed >= amount, "Allowance exceeded");

    allowance[from][msg.sender] = allowed - amount;
    emit Approval(from, msg.sender, allowance[from][msg.sender]);

    _transfer(from, to, amount);
    return true;
}

5.4 Spending Constraint Enforcement

Code Proof
require(allowed >= amount, "Allowance exceeded");

5.5 Deterministic Allowance Reduction

Code Proof
allowance[from][msg.sender] = allowed - amount;

5.6 Event-Based Delegation Tracking

Code Proof
emit Approval(from, msg.sender, allowance[from][msg.sender]);

5.7 Absence of Advanced Approval Mechanisms

  • No increaseAllowance function
  • No decreaseAllowance function
  • No permit-based approvals (EIP-2612)

5.8 Security Considerations

The allowance system follows the standard ERC-20 model without additional protections. Approvals overwrite previous values, users must manually reset allowances if needed, and no built-in mitigation exists for approval race conditions.

AXION’s allowance mechanism provides a standard, deterministic, and fully on-chain delegation system, where all third-party spending permissions are explicitly defined, strictly enforced, and transparently tracked.
